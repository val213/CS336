## 2 过滤 Common Crawl 数据

虽然大型语言模型主要使用互联网数据进行训练，但大多数研究人员并不会自己构建网络爬虫来收集训练数据。他们通常使用公开可用的爬虫数据集。最流行的公开网络爬虫数据集来自 Common Crawl（通用爬虫）——这是一家非营利组织，提供免费的网页语料库，其数据库"包含跨越17年、超过2500亿个网页"[^1]。

然而，将 Common Crawl（CC）转储数据转化为可用于语言模型训练的数据需要大量工作。例如，网页的原始数据是 HTML 格式的，我们需要从中提取纯文本。此外，许多页面可能质量低下、存在完全或近似重复、包含有害内容，或含有敏感信息，我们可能需要过滤掉这些页面，或从数据集中删除其中不良内容。在本次作业中，我们将建立一个数据处理流水线，执行上述多个步骤，将原始互联网数据转化为可用于语言模型训练的训练集。

[^1]: https://commoncrawl.org/

### 2.1 查看原始数据

在实现任何功能之前，查看原始数据并对其有所了解总是很有价值的。Common Crawl 数据以三种格式提供：

- **WARC（Web ARChive format，网络存档格式）文件**：包含 CC 的原始数据，其中包括页面 ID 和 URL、元数据及 HTTP 请求详情（如请求的日期和时间、服务器 IP 地址），以及页面的原始内容（如 HTML）。
- **WAT（Web Archive Transformation，网络存档转换）文件**：包含从 WARC 文件中提取并以 JSON 对象格式存储的高级元数据。例如，对于 HTML 页面，这包括该页面的链接列表和页面标题。
- **WET（Web Extracted Text，网络提取文本）文件**：包含从原始 HTML 页面中提取的纯文本内容。

在下面的题目中，我们将查看一个 WARC 文件及其对应的 WET 文件。这些文件来自2025年4月的爬取结果，下载方式如下：

```bash

# 下载对应的 WET 文件
$ wget https://data.commoncrawl.org/crawl-data/CC-MAIN-2025-18/segments/1744889135610.12/wet/CC-MAIN-20250417135010-20250417165010-00065.warc.wet.gz
```

在 Together 集群上，这些文件可在以下路径找到：

- `/data/CC/example.warc.gz`
- `/data/CC/example.warc.wet.gz`

> **警告**：这些文件包含完全未经过滤的互联网页面，可能含有大量潜在有害内容。如果你看到不想阅读的文档，可以继续翻页，跳过到下一个。

### 2.2 HTML 转纯文本

正如你从查看 WARC 和 WET 文件中意识到的，从 HTML 中提取文本是一项具有挑战性的工作。通常，任何提取程序都会寻找 HTML 中的可见内容（例如 `<p>` 标签，它们应当包含文本块）。但这仍然可能提取出比我们在浏览器中打开页面时所看到的主要内容多得多的内容。例如，打开 StackOverflow 时，主要内容在问题和答案中，但从技术上讲，菜单选项、指向其他 StackExchange 无关页面的链接、页脚、注册或登录链接——这些都是可见文本，要可靠地将它们与页面主要内容区分开来是非常具有挑战性的。

许多工具实现了文本提取流水线。在本次作业中，我们将使用 Resiliparse[^2] 库来进行文本提取。Resiliparse 还将帮助解决一个更基本的问题：检测包含原始内容的字节串的文本编码。虽然网络上大多数页面使用 UTF-8 编码（根据维基百科，占 98.2%），但我们的文本提取流水线也应该对其他编码具有鲁棒性。

> **注意**：我们建议使用 FastWARC 库来遍历每个 WARC 文件中的记录。具体来说，以下类可能会有所帮助：

```python
from fastwarc.warc import ArchiveIterator, WarcRecordType
```

[^2]: https://resiliparse.chatnoir.eu/en/stable/index.html

### 2.3 语言识别

### 2.5 有害内容过滤

未经过滤的网络转储数据中包含大量我们不希望语言模型在推理时重复的文本。其中一些训练样本甚至可能出乎意料地来自维基百科等通常无害的网站——例如，用户在多个页面上留下的评论可能相当具有毒性。

尽管明确界定"有害内容"的范围几乎是不可能的，但许多数据过滤流水线仍然会对主要包含有害内容的页面进行过滤。

识别此类内容的方法有很多，包括统计黑名单词汇的频率，或基于人工评级标签构建简单分类器。在作业的这一部分，我们将重点识别两大类有害内容：

### 2.6 质量规则过滤

即使在按语言过滤并去除有害内容之后，剩余页面中仍然有相当大比例对语言模型训练来说质量较低。同样，"质量"并不容易定义，但浏览 Common Crawl 的示例有助于识别低质量内容的典型案例，例如：

- 带有付费墙内容的页面
- 死链的占位符页面
- 登录、注册或联系表单页面
- 主要包含非文本内容（在文本提取过程中丢失）的页面（例如照片、视频）

## 3 去重处理

网络上存在大量重复内容。有些页面是彼此的完全副本——例如存档页面，或由标准工具生成的默认页面（如流行网络服务器的404页面）。但大多数重复发生在更细粒度的层面。例如，考虑 Stack Overflow 上的所有问题页面：虽然每个页面都有独特的内容（如问题本身、评论和回答），但所有页面也有大量冗余内容，例如页眉、菜单选项和页脚，这些内容在渲染所有此类页面时会以完全相同的形式重复出现。

在本节的第一部分，我们将处理这种精确重复，随后学习如何处理近似重复。

---

### 3.1 精确行去重

一种简单的精确重复去重方法是只保留文档中在语料库中唯一出现的行。事实证明，这足以消除大量冗余，例如上面提到的页眉和菜单选项。在更简单的情况下，删除在其他地方精确重复的行后，我们通常只剩下每个页面的独特主要内容（例如 StackOverflow 上的问题和回答）。

为此，我们可以对语料库进行一次遍历，统计每行出现的次数。然后在第二次遍历中，通过只保留唯一的行来重写每个文档。

简单实现时，用于保存计数的数据结构可能需要与存储语料库中所有唯一行所需的内存一样多。一个简单的内存优化技巧是使用行的哈希值作为键，使键具有固定大小（而不是依赖于行的长度）。你现在将实现这种简单的去重方法。

### 3.2 MinHash + LSH 文档去重

精确去重对于删除多个网页中逐字重复的内容很有用，但无法处理文档内容稍有差异的情况。例如，考虑软件许可证文档——许可证文档通常由需要填写年份和软件作者名称的模板生成。因此，一个 MIT 许可证项目的许可证文件与另一个 MIT 许可证项目的许可证文件包含大量相同内容，但它们并不是完全相同的副本。为了删除这类重复的、大部分由模板生成的内容，我们需要使用模糊去重（fuzzy deduplication）。为了高效执行模糊文档级去重，我们将使用 MinHash 结合局部敏感哈希（Locality Sensitive Hashing，LSH）[^6]。

为了执行模糊去重，我们将使用一种特定的文档相似度度量：文档 n-gram 之间的 Jaccard 相似度。集合 S 和 T 之间的 Jaccard 相似度定义为 |S ∩ T| / |S ∪ T|。简单实现模糊去重的方法是将每个文档表示为一组 n-gram，计算所有文档对之间的 Jaccard 相似度，并将超过特定 Jaccard 相似度阈值的对标记为重复。然而，这种方法对于大型文档集合（如 Common Crawl）来说是不切实际的。此外，简单存储一组 n-gram 将比文档本身占用更多的内存。

#### MinHash（最小哈希）

为了解决内存问题，我们将文档的 n-gram 集合表示替换为签名（signature）。具体来说，我们希望构建这样的签名：通过比较两个文档的签名，我们能得到对这两个文档 n-gram 集合之间 Jaccard 相似度的近似估计。MinHash 签名满足这些性质。

要计算文档 n-gram 集合 S = {s1, ..., sn} 的 MinHash 签名，我们需要 k 个不同的哈希函数 h1, ..., hk [^7]。每个哈希函数将一个 n-gram 映射到一个整数。给定哈希函数 hi，n-gram 集合 S 的 minhash 为：

    minhash(hi, S) = min(hi(s1), hi(s2), ..., hi(sn))

文档 n-gram 集合 S 的签名是 R^k 中的一个向量，其中第 i 个元素包含 S 在随机哈希函数 hi 下的 minhash 值：

    [minhash(h1, S), minhash(h2, S), ..., minhash(hk, S)]

事实证明，对于两个文档的 n-gram 集合 S1 和 S2，集合之间的 Jaccard 相似度可以用签名中具有相同 minhash 值的列的比例来近似（证明见 Leskovec et al., 2014 第3章第3.3.3节）。例如，给定文档签名 [1, 2, 3, 2] 和 [5, 2, 3, 4]，n-gram 集合之间的 Jaccard 相似度近似为 2/4，因为这两个签名的第二列和第三列具有相同的 minhash 值。

#### 局部敏感哈希（LSH）

虽然 MinHash 给出了一种内存高效的文档表示，且保留了任意文档对之间的期望相似度，但我们仍然需要比较所有文档对来找出最相似的文档。LSH 提供了一种高效地将可能具有高相似度的文档分组的方法。

为了将 LSH 应用于我们的文档签名（此时为 R^k 中的向量），我们将签名分成 b 个频带（band），每个频带包含 r 个 minhash 值，满足 k = b * r。例如，如果我们有100个元素的文档签名（由100个随机哈希函数生成），我们可以将其分成2个各含50个 minhash 的频带，或4个各含25个 minhash 的频带，或50个各含2个 minhash 的频带，等等。

如果两个文档在某个特定频带中具有相同的哈希值，它们将被聚类到同一个桶（bucket）中，并被视为候选重复项。因此，在签名数量固定的情况下，增加频带数量将提高召回率，降低精确率。

**具体示例**：假设文档 D1 的 MinHash 签名为 [1, 2, 3, 4, 5, 6]，文档 D2 的签名为 [1, 2, 3, 5, 1, 2]。若使用3个频带，每个频带含2个 minhash，则：
- D1 的第一个频带为 [1, 2]，第二个为 [3, 4]，第三个为 [5, 6]
- D2 的第一个频带为 [1, 2]，第二个为 [3, 5]，第三个为 [1, 2]

由于第一个频带的哈希值匹配（两个文档都是 [1, 2]），D1 和 D2 在第一个频带下会被聚集在一起。其他频带哈希值不匹配，因此它们不会在其他频带下聚集。但由于文档在至少一个频带中被聚集在一起，无论其他频带是否匹配，它们都会被视为候选重复项。

一旦确定了候选重复项，我们可以用多种方式处理它们。例如，我们可以计算所有候选重复项之间的 Jaccard 相似度，并将超过设定阈值的对标记为重复项。

最后，我们在桶间对重复文档进行聚类。例如，假设文档 A 和 B 在一个桶中匹配，且其真实 Jaccard 相似度高于我们的阈值；文档 B 和 C 在另一个桶中匹配，且其真实 Jaccard 相似度也高于我们的阈值。那么，我们会将文档 A、B 和 C 视为一个集群，并随机删除每个集群中除一个之外的所有文档。

[^6]: 有关 LSH 和 MinHash 的更深入介绍，请参阅 Leskovec et al. [2014] 的第3章，该章节可在 infolab.stanford.edu/ullman/mmds/ch3n.pdf 在线获取。
[^7]: 这 k 个不同的哈希函数可以属于同一哈希函数族，但使用不同的随机种子。例如，MurmurHash3 是一族哈希函数，使用特定种子可以实例化该族中的特定哈希函数。

```python
import concurrent.futures
import os
from tqdm import tqdm

def process_single_wet_file(input_path: str, output_path: str):
    # TODO: 读取输入路径，处理输入，并将输出写入 output_path
    return output_path

# 设置执行器
num_cpus = len(os.sched_getaffinity(0))
executor = concurrent.futures.ProcessPoolExecutor(max_workers=num_cpus)

wet_filepaths = ["a.warc.wet.gz", "b.warc.wet.gz", "c.warc.wet.gz"]
output_directory_path = "/path/to/output_directory/"

futures = []
for wet_filepath in wet_filepaths:
    # 对每个 warc.wet.gz 文件路径，向执行器提交任务并获取 future 对象
    wet_filename = str(pathlib.Path(wet_filepath).name)
    future = executor.submit(
        process_single_wet_file,
        wet_filepath,
        os.path.join(output_directory_path, wet_filename)
    )
    # 存储 future 对象
    futures.append(future)

# 使用进度条迭代已完成的 future 对象
for future in tqdm(
    concurrent.futures.as_completed(futures),
    total=len(wet_filepaths),
):
    output_file = future.result()
    print(f"Output file written: {output_file}")
```

如你所见，使用 submitit 与使用内置 `concurrent.futures` API 的代码非常相似。主要区别在于：(1) 需要配置 submitit 执行器参数，以便它知道在哪里提交 Slurm 作业以及每个作业的资源规格；(2) 使用 `executor.batch()` 将所有 Slurm 作业分组为单个作业"数组"（而不是 `len(wet_filepaths)` 个单独的作业），这可以减少 Slurm 调度器的负载；(3) 在收集结果时使用 `submitit.helpers.as_completed`。

我们还建议使用 FastWARC 库来遍历每个 WET 文件中的记录，以及使用 tldextract 库从 URL 中提取域名用于过滤。特别是，以下类可能会有所帮助：

```python

```

在对语言模型训练数据进行分词（tokenization）之前，我们需要先对其进行分词处理。使用 `transformers` 库中的 GPT-2 分词器将你过滤后的数据编码为整数 ID 序列，用于训练语言模型。不要忘记在每个文档之后添加 GPT-2 的序列结束标记 `<|endoftext|>`。以下是一些起始代码：

```python
import multiprocessing
import numpy as np
from tqdm import tqdm
from transformers import AutoTokenizer

input_path = "path/to/your/filtered/data"
output_path = "path/to/your/tokenized/data"

tokenizer = AutoTokenizer.from_pretrained("gpt2")

def tokenize_line_and_add_eos(line):
    return tokenizer.encode(line) + [tokenizer.eos_token_id]

with open(input_path) as f:
    lines = f.readlines()

pool = multiprocessing.Pool(multiprocessing.cpu_count())
chunksize = 100
results = []
for result in tqdm(
    pool.imap(tokenize_line_and_add_eos, lines, chunksize=chunksize),
    total=len(lines),
    desc="Tokenizing lines"
):
    results.append(result)

pool.close()
pool.join()

# 将 ID 列表展平并转换为 numpy 数组
all_ids = [token_id for sublist in results for token_id in sublist]
print(f"Tokenized and encoded {input_path} into {len(all_ids)} tokens")
ids_array = np.array(all_ids, dtype=np.uint16)
ids_array.tofile(output_path)
```
