性能测量是微妙的——一些常见陷阱可能导致我们测量的不是我们想要的。对于基准测试 GPU 代码，一个注意事项是 CUDA 调用是**异步的**。当你调用 CUDA 内核（如 `torch.matmul`）时，函数调用会在矩阵乘法完成之前将控制权返回给你的代码。这样 CPU 可以在 GPU 计算矩阵乘法的同时继续运行。另一方面，这意味着简单地测量 `torch.matmul` 调用返回所需的时间并不能告诉我们 GPU 实际运行矩阵乘法需要多长时间。在 PyTorch 中，我们可以调用 `torch.cuda.synchronize()` 来等待所有 GPU 内核完成，从而获得更准确的 CUDA 内核运行时间测量。

### 1.1.4 Nsight Systems 分析器

端到端基准测试不能告诉我们模型在前向和反向传播中在哪里花费时间和内存，因此不能暴露具体的优化机会。要知道程序在每个组件（如函数）中花费多少时间，我们可以使用分析器（profiler）。执行分析器通过在函数开始和结束运行时插入守卫来检测代码，从而可以在函数级别给出详细的执行统计信息（如调用次数、平均耗时、在此函数上花费的累计时间等）。

标准 Python 分析器（如 CProfile）无法分析 CUDA 内核，因为这些内核是在 GPU 上异步执行的。幸运的是，NVIDIA 提供了一个我们可以通过 CLI `nsys` 使用的分析器，我们已经为你安装好了。在作业的这部分，你将使用 `nsys` 来分析 Transformer 模型的运行时。使用 `nsys` 很简单：我们可以在你之前编写的 Python 脚本前面加上 `nsys profile`。例如，你可以分析脚本 `benchmark.py` 并将输出写入文件 `result.nsys.rep`：

```
~$ uv run nsys profile -o result python benchmark.py
```

然后你可以在本地机器上使用 NVIDIA Nsight Systems 桌面应用程序查看分析结果。在 CUDA API 行中选择特定的 CUDA API 调用（在 CPU 上），将在 CUDA HW 行中高亮显示所有对应的内核执行（在 GPU 上）。

我们鼓励你尝试 `nsys profile` 的各种命令行选项。值得注意的是，你可以通过 `--python-backtrace=cuda` 获取每个 CUDA API 调用的 Python 回溯，但这可能引入开销。你还可以用 NVTX 范围注解你的代码，这些范围将出现在分析的 NVTX 行中，捕获所有 CUDA API 调用和相关的内核执行。特别是，你应该**使用 NVTX 范围来忽略基准测试脚本中的预热步骤**（通过在分析中对 NVTX 行应用过滤器）。你还可以隔离哪些内核负责模型的前向和反向传播，甚至可以通过如下注解来隔离自注意力层中不同部分负责的内核：

### 1.1.5 混合精度（Mixed Precision）

到目前为止，我们一直使用 FP32 精度运行——所有模型参数和激活值的数据类型都是 `torch.float32`。然而，现代 NVIDIA GPU 包含专门的 GPU 核心（Tensor Cores）用于加速低精度的矩阵乘法。例如，NVIDIA A100 规格表显示其 FP32 的最大吞吐量为 19.5 TFLOP/秒，而 FP16（半精度浮点）或 BF16（Brain Float）的最大吞吐量显著更高，为 312 TFLOP/秒。因此，使用低精度数据类型应该有助于加速训练和推理。

然而，简单地将模型转换为低精度格式可能会降低模型准确性。例如，实践中许多梯度值通常太小，无法在 FP16 中表示，因此在使用 FP16 精度训练时会变为零。为了解决这个问题，使用 FP16 训练时通常会使用**损失缩放**（loss scaling）——损失简单地乘以一个缩放因子，增加梯度幅度使其不会刷新为零。此外，FP16 的动态范围比 FP32 低，这可能导致溢出表现为 NaN 损失。完全的 bfloat16 训练通常更稳定（因为 BF16 与 FP32 具有相同的动态范围），但仍可能影响最终模型性能。

为了利用低精度数据类型的加速，通常使用**混合精度训练**。在 PyTorch 中，这通过 `torch.autocast` 上下文管理器实现。在这种情况下，某些操作（如矩阵乘法）以低精度数据类型执行，而其他需要 FP32 完整动态范围的操作（如累加和归约）保持不变。例如：

```python
model : torch.nn.Module = ...  # 例如你的 Transformer 模型
dtype : torch.dtype = ...      # 例如 torch.float16
x : torch.Tensor = ...         # 输入数据

with torch.autocast(device="cuda", dtype=dtype):
    y = model(x)
```

如上所述，即使被累加的张量本身已被降精度转换，保持累加在更高精度通常也是个好主意。以下练习将帮助你建立这方面的直觉。

### 1.1.6 内存分析（Profiling Memory）

到目前为止，我们一直在关注计算性能。现在我们将转向内存——语言模型训练和推理中的另一个主要资源。PyTorch 也自带了一个强大的内存分析器，可以跟踪随时间变化的内存分配。

要使用内存分析器，你可以如下修改基准测试脚本：

```python
... # 基准测试脚本中的预热阶段

# 开始记录内存历史。
torch.cuda.memory._record_memory_history(max_entries=1000000)

... # 基准测试脚本中你想要分析的部分

# 保存 pickle 文件，由 PyTorch 的在线工具加载。
torch.cuda.memory._dump_snapshot("memory_snapshot.pickle")

# 停止记录历史。
torch.cuda.memory._record_memory_history(enabled=None)
```

这将输出一个 `memory_snapshot.pickle` 文件，你可以将其加载到以下在线工具中：https://pytorch.org/memory_viz 。该工具可让你查看整体内存使用时间线以及每个单独的内存分配及其大小和指向分配源代码的堆栈跟踪。使用方法：在浏览器中打开上述链接，然后将 Pickle 文件拖放到页面上。

### 1.3.1 示例 - 加权求和（Weighted Sum）

为了介绍你需要了解的 Triton 知识以及它如何与 PyTorch 互操作，我们将演练一个"加权求和"操作的示例内核。有关 Triton 的更多资源，请参见 Triton 的教程。我们注意到这些教程不使用新的、方便的块指针（block pointer）抽象，我们将在下面演示。

给定输入矩阵 X，我们将其元素乘以列向量权重 w，并对每行求和，得到 X 和 w 的矩阵向量积。我们将先完成此操作的前向传播，然后为反向传播编写 Triton 内核。

**前向传播** 我们内核的前向传播就是以下广播内积：

```python
def weighted_sum(x, weight):
    # 假设 x 的 n 维形状为 [..., D]，weight 的1维形状为 [D]
    return (weight * x).sum(axis=-1)
```

编写 Triton 内核时，我们让每个程序实例（可能并行运行）计算 x 的一个行块（tile）的加权求和，并将相应的标量输出写入输出张量。在 Triton 中，一个程序实例是一组线程块，所有线程运行相同的程序，这些线程块可以在 GPU 上并行运行。我们不接受张量作为参数，而是接受指向其第一个元素的指针，以及告诉我们如何沿各轴移动的步幅。

我们使用块指针抽象 `tl.make_block_ptr` 来大大简化指针算术。参考图1了解块指针的分块和推进方式。

**前向传播 Triton 内核：**

```python
import triton
import triton.language as tl

@triton.jit
def weighted_sum_fwd(
    x_ptr, weight_ptr,          # 输入指针
    output_ptr,                  # 输出指针
    x_stride_row, x_stride_dim, # 步幅告诉我们如何在张量的每个轴上移动一个元素
    weight_stride_dim,           # 可能为1
    output_stride_row,           # 可能为1
    ROWS, D,
    ROWS_TILE_SIZE: tl.constexpr, D_TILE_SIZE: tl.constexpr, # 块形状必须在编译时已知
):
    # 每个实例将计算 x 的一个行块的加权求和。
    # `tl.program_id` 给我们一种方式来检查我们在哪个线程块中运行
    row_tile_idx = tl.program_id(0)

    # 块指针给我们一种方式来选择 ND 内存区域并移动我们的选择。
    # 块指针必须知道：
    # - 张量第一个元素的指针
    # - 张量的整体形状以处理越界访问
    # - 每个维度的步幅以正确使用内存布局
    # - 起始块的 ND 坐标，即"offsets"
    # - 每次 load/store 使用的块形状
    # - 内存中从主维度到次维度的维度顺序

    x_block_ptr = tl.make_block_ptr(
        x_ptr,
        shape=(ROWS, D,),
        strides=(x_row_stride, x_stride_dim),
        offsets=(row_tile_idx * ROWS_TILE_SIZE, 0),
        block_shape=(ROWS_TILE_SIZE, D_TILE_SIZE),
        order=(1, 0),
    )

    weight_block_ptr = tl.make_block_ptr(
        weight_ptr,
        shape=(D,),
        strides=(weight_stride_dim,),
        offsets=(0,),
        block_shape=(D_TILE_SIZE,),
        order=(0,),
    )

    output_block_ptr = tl.make_block_ptr(
        output_ptr,
        shape=(ROWS,),
        strides=(output_stride_row,),
        offsets=(row_tile_idx * ROWS_TILE_SIZE,),
        block_shape=(ROWS_TILE_SIZE,),
        order=(0,),
    )

    # 初始化一个缓冲区用于写入
    output = tl.zeros((ROWS_TILE_SIZE,), dtype=tl.float32)

    for i in range(tl.cdiv(D, D_TILE_SIZE)):
        # 加载当前块指针
        # 由于 ROWS_TILE_SIZE 可能不整除 ROWS，D_TILE_SIZE 可能不整除 D，
        # 我们需要对两个维度进行边界检查
        row = tl.load(x_block_ptr, boundary_check=(0, 1), padding_option="zero")
        weight = tl.load(weight_block_ptr, boundary_check=(0,), padding_option="zero")

        # 计算行的加权求和
        output += tl.sum(row * weight[None, :], axis=1)

        # 将指针移动到下一个块
        x_block_ptr = x_block_ptr.advance((0, D_TILE_SIZE))
        weight_block_ptr = weight_block_ptr.advance((D_TILE_SIZE,))

    # 将输出写入输出块指针（每行一个标量）
    tl.store(output_block_ptr, output, boundary_check=(0,))
```

现在让我们将这个内核包装在 PyTorch Autograd 函数中，使其能与 PyTorch 互操作（即接受张量作为输入，输出张量，稍后还能在反向传播中与 autograd 引擎配合工作）：

```python
class WeightedSumFunc(torch.autograd.Function):
    @staticmethod
    def forward(ctx, x, weight):
        # 缓存 x 和 weight 用于反向传播，此时我们
        # 只接收关于输出张量的梯度，
        # 需要计算关于 x 和 weight 的梯度。
        D, output_dims = x.shape[-1], x.shape[:-1]

        # 将输入张量重塑为 2D
        input_shape = x.shape
        x = rearrange(x, "... d -> (...) d")

        ctx.save_for_backward(x, weight)

        assert len(weight.shape) == 1 and weight.shape[0] == D, "维度不匹配"
        assert x.is_cuda and weight.is_cuda, "期望 CUDA 张量"
        assert x.is_contiguous(), "我们的指针算术假设 x 是连续的"

        ctx.D_TILE_SIZE = triton.next_power_of_2(D) // 16
        ctx.ROWS_TILE_SIZE = 16
        ctx.input_shape = input_shape

        # 需要初始化空的结果张量。注意这些元素不一定是0！
        y = torch.empty(output_dims, device=x.device)

        # 启动内核，在1D网格中有 n 个实例
        n_rows = y.numel()
        weighted_sum_fwd[(cdiv(n_rows, ctx.ROWS_TILE_SIZE),)](
            x, weight,
            y,
            x.stride(0), x.stride(1),
            weight.stride(0),
            y.stride(0),
            ROWS=n_rows, D=D,
            ROWS_TILE_SIZE=ctx.ROWS_TILE_SIZE, D_TILE_SIZE=ctx.D_TILE_SIZE,
        )

        return y.view(input_shape[:-1])
```

注意当我们用 `weighted_sum_fwd[(cdiv(n_rows, ctx.ROWS_TILE_SIZE),)]` 调用 Triton 内核时，我们通过传递元组 `(cdiv(n_rows, ctx.ROWS_TILE_SIZE),)` 定义了一个所谓的"启动网格"（launch grid）。然后我们可以在内核中用 `tl.program_id(0)` 访问线程块索引。

**反向传播**  由于我们定义了自己的内核，我们还需要编写自己的反向函数。

在前向传播中，我们得到层的输入并计算输出。在反向传播中，回忆一下我们将得到关于输出的目标梯度，需要计算关于每个输入的梯度。在我们的情况下，操作的输入是矩阵 $x \in \mathbb{R}^{n \times h}$ 和权重向量 $w \in \mathbb{R}^h$。简写为 $f(x, w)$，其值域为 $\mathbb{R}^n$。假设我们已知 $\nabla_{f(x,w)} L$，即损失 L 关于层输出的梯度，我们可以应用多变量链式法则得到关于 x 和 w 的梯度表达式：

$$(\nabla_x L)_{ij} = w_j \cdot (\nabla_{f(x,w)} L)_i$$

$$(\nabla_w L)_j = \sum_{i=1}^{n} x_{ij} \cdot (\nabla_{f(x,w)} L)_i$$

这给出了计算反向传播的简单公式。对于 x 的反向步骤，我们应用上式取 w 和 $\nabla_{f(x,w)}$ 的外积。对于 w 的反向步骤，我们将输入梯度乘以对应的输出行。

**反向传播 Triton 内核**（含块指针设置和 grad_x、grad_weight 计算——完整代码见原文）

计算 $\nabla_x$ 很简单，我们将结果写入输出张量的适当块。然而计算 $\nabla_w$ 更具挑战性。每个内核实例负责 x 的一个行块，但我们现在需要跨 x 的行求和。我们不在反向传播内核中直接做这个求和，而是假设 `partial_grad_weight_ptr` 包含一个 `n_row_tiles × H` 的矩阵，第一个维度只在 x 的一个行块内进行归约。在内核外部，我们使用 `torch.sum` 来汇总每个行块的结果[^1]。

[^1]: 当然，我们也可以为此编写自己的内核。

`autograd.Function` 的最后部分则相对简单：

```python
class WeightedSumFunc(torch.autograd.Function):
    @staticmethod
    def forward(ctx, x, weight):
        # ... （之前已定义）

    @staticmethod
    def backward(ctx, grad_out):
        x, weight = ctx.saved_tensors
        ROWS_TILE_SIZE, D_TILE_SIZE = ctx.ROWS_TILE_SIZE, ctx.D_TILE_SIZE
        n_rows, D = x.shape

        partial_grad_weight = torch.empty(
            (cdiv(n_rows, ROWS_TILE_SIZE), D), device=x.device, dtype=x.dtype
        )
        grad_x = torch.empty_like(x)

        weighted_sum_backward[(cdiv(n_rows, ROWS_TILE_SIZE),)](
            x, weight,
            grad_out,
            grad_x, partial_grad_weight,
            x.stride(0), x.stride(1),
            weight.stride(0),
            grad_out.stride(0),
            grad_x.stride(0), grad_x.stride(1),
            partial_grad_weight.stride(0), partial_grad_weight.stride(1),
            NUM_ROWS=n_rows, D=D,
            ROWS_TILE_SIZE=ROWS_TILE_SIZE, D_TILE_SIZE=D_TILE_SIZE,
        )
        grad_weight = partial_grad_weight.sum(axis=0)
        return grad_x, grad_weight
```

最后，我们可以得到一个像 `torch.nn.functional` 中实现的函数那样工作的函数：

```python
f_weightedsum = WeightedSumFunc.apply
```

注意输出张量附带的 `grad_fn`——这表明 PyTorch 知道在反向传播中当此张量出现在计算图中时该调用什么。这就完成了加权求和操作的 Triton 实现。

### 1.3.2 FlashAttention-2 前向传播

你将用一个显著改进的 Triton 实现来替换你的 PyTorch 注意力实现，该实现遵循 FlashAttention-2 [Dao, 2023]。FlashAttention-2 使用一些技巧来分块计算前向传播，这允许高效的内存访问模式并避免在全局内存中实例化完整的注意力矩阵。

在进入本节之前，我们**强烈建议**至少阅读原始 FlashAttention 论文 [Dao et al., 2022]，它将为你提供 FlashAttention 高效注意力核心技术的直觉：以在线方式跨块计算 softmax（[Milakov and Gimelshein, 2018] 提出的技术）。我们还推荐查看 He [2022] 以获取更多关于 GPU 如何实际执行 PyTorch 代码的直觉。

**理解朴素注意力的低效性。** 回忆注意力的前向传播（暂时忽略掩码）可以写为：

$$S = QK^\top / \sqrt{d}$$
$$P_{ij} = \text{softmax}_j(S)_{ij}$$
$$O = PV$$

标准反向传播为：

$$dV = P^\top dO$$
$$dP = dOV^\top$$
$$dS_i = d\text{softmax}(dP_i) = (\text{diag}(P_i) - P_i P_i^\top) dP_i$$
$$dQ = dSK / \sqrt{d}$$
$$dK = dS^\top Q / \sqrt{d}$$

如我们所见，反向传播依赖于前向传播中一些非常大的激活值。例如，计算(7)中的 dV 需要 P，其形状为 `(batch_size, n_heads, seq_len, seq_len)`——这个激活矩阵的大小与序列长度呈二次关系，解释了我们之前在大序列长度基准测试注意力时遇到的内存问题。

FlashAttention 的主要目标是避免将注意力矩阵读写到 HBM，以减少 IO 和峰值内存开销。我们通过三种技术实现：**分块**（tiling）、**重计算**（recomputation）和**算子融合**（operator fusion）。

**分块（Tiling）。** 为了避免将注意力矩阵读写到 HBM，我们在不访问整个输入的情况下计算 softmax 归约。具体来说，我们将注意力计算重组为将输入分成块，并对输入块进行多次遍历，从而增量地执行 softmax 归约。

**重计算（Recomputation）。** 我们避免在 HBM 中存储形状为 `(batch_size, n_heads, seq_len, seq_len)` 的大型中间注意力矩阵。相反，我们在 HBM 中保存某些"激活检查点"，然后在反向传播期间重计算部分前向传播，以获取计算梯度所需的其他激活值。FlashAttention-2 还存储注意力分数的 logsumexp L：

$$L_i = \log\left(\sum_j \exp(S_{ij})\right)$$

在我们的最终内核中我们将以在线方式计算这个值，但最终结果应该是相同的。

**算子融合（Operator fusion）。** 最后，我们通过在单个内核中执行所有操作来避免注意力矩阵和其他中间激活值的重复内存 IO——这被称为算子融合或内核融合。

**使用重计算的反向传播。** 使用 L，我们可以做适当的重计算并高效计算反向传播。在开始反向传播之前，我们预计算 $D = \text{rowsum}(O \circ dO)$（其中 $\circ$ 是逐元素乘法）。有了 L 和 D 向量，反向传播可以在不需要 softmax 的情况下计算：

$$S = QK^\top / \sqrt{d}$$
$$P_{ij} = \exp(S_{ij} - L_i)$$
$$dV = P^\top dO$$
$$dP = dOV^\top$$
$$dS_{ij} = P_{ij} \circ (dP_{ij} - D_i)$$
$$dQ = dSK / \sqrt{d}$$
$$dK = dS^\top Q / \sqrt{d}$$

**FlashAttention-2 前向传播详细说明。**

在 FlashAttention-2 中，我们使用**在线 softmax**。使用下标 i 表示当前查询块，上标 (j) 表示当前键块。查询维度的块大小为 $B_q$，键维度为 $B_k$。我们不沿隐藏维度 d 分块。

我们还保持一些行级的运行值：$m_i^{(j)} \in \mathbb{R}^{B_q}$ 和 $l_i^{(j)} \in \mathbb{R}^{B_q}$。$m_i^{(j)}$ 是运行最大值，用于数值稳定地计算 softmax。$l_i^{(j)}$ 是 softmax 分母的运行代理。

**算法1：FlashAttention-2 前向传播**

输入：$Q \in \mathbb{R}^{N_q \times d}$，$K, V \in \mathbb{R}^{N_k \times d}$，块大小 $B_q, B_k$

将 Q 分成 $T_q = \lceil N_q / B_q \rceil$ 个块 $Q_1, \ldots, Q_{T_q}$，大小 $B_q \times d$
将 K, V 分成 $T_k = \lceil N_k / B_k \rceil$ 个块 $K^{(1)}, \ldots, K^{(T_k)}$ 和 $V^{(1)}, \ldots, V^{(T_k)}$，大小 $B_k \times d$

**for** $i = 1, \ldots, T_q$ **do**
  - 从全局内存加载 $Q_i$
  - 初始化 $O_i^{(0)} = 0$，$l_i^{(0)} = 0$，$m_i^{(0)} = -\infty$
  - **for** $j = 1, \ldots, T_k$ **do**
    - 从全局内存加载 $K^{(j)}, V^{(j)}$
    - 计算注意力分数块：$S_i^{(j)} = Q_i (K^{(j)})^\top / \sqrt{d}$
    - 计算 $m_i^{(j)} = \max(m_i^{(j-1)}, \text{rowmax}(S_i^{(j)}))$
    - 计算 $\tilde{P}_i^{(j)} = \exp(S_i^{(j)} - m_i^{(j)})$
    - 计算 $l_i^{(j)} = \exp(m_i^{(j-1)} - m_i^{(j)}) l_i^{(j-1)} + \text{rowsum}(\tilde{P}_i^{(j)})$
    - 计算 $O_i^{(j)} = \text{diag}(\exp(m_i^{(j-1)} - m_i^{(j)})) O_i^{(j-1)} + \tilde{P}_i^{(j)} V^{(j)}$
  - **end for**
  - 计算 $O_i = \text{diag}(l_i^{(T_k)})^{-1} O_i^{(T_k)}$
  - 计算 $L_i = m_i^{(T_k)} + \log(l_i^{(T_k)})$
  - 将 $O_i$ 写入全局内存作为 O 的第 i 块
  - 将 $L_i$ 写入全局内存作为 L 的第 i 块

**end for**

返回输出 O 和 logsumexp L。

**Triton 提示与技巧：**
- 你可以在 Triton 中使用 `tl.device_print` 进行调试打印。有一个设置 `TRITON_INTERPRET=1` 可在 CPU 上运行 Triton 解释器，但我们发现它有 bug。
- 定义块指针时，确保偏移量正确，块偏移量乘以了适当的块大小。

### 1.3.4 可选：Triton 反向传播

如果你有兴趣获得更多 Triton 实践经验和/或获得更快的排行榜提交，我们提供分块 FlashAttention-2 反向传播（算法2），你可以在 Triton 中实现。关键技巧是计算 P 两次——一次用于 dQ 的反向传播，另一次用于 dK 和 dV，这让我们可以跳过跨线程块的同步。

**算法2：分块 FlashAttention-2 反向传播**

输入：$Q, O, dO \in \mathbb{R}^{N_q \times d}$，$K, V \in \mathbb{R}^{N_k \times d}$，$L \in \mathbb{R}^{N_q}$，块大小 $B_q, B_k$

计算 $D = \text{rowsum}(dO \circ O)$

**for** $j = 1, \ldots, T_k$ **do**
  - 加载 $K^{(j)}, V^{(j)}$
  - 初始化 $dK^{(j)} = dV^{(j)} = 0$
  - **for** $i = 1, \ldots, T_q$ **do**
    - 加载 $Q_i, O_i, dO_i, dQ_i$
    - 计算 $S_i^{(j)} = Q_i (K^{(j)})^\top / \sqrt{d}$
    - 计算 $P_i^{(j)} = \exp(S_i^{(j)} - L_i)$
    - 计算 $dV^{(j)} += (P_i^{(j)})^\top dO_i$
    - 计算 $dP_i^{(j)} = dO_i V_j^\top$
    - 计算 $dS_i^{(j)} = P_i^{(j)} \circ (dP_i^{(j)} - D_i) / \sqrt{d}$
    - 从全局内存加载 $dQ_i$，更新 $dQ_i += dS_i^{(j)} K^{(j)}$，写回全局内存（必须是原子操作！）
    - 计算 $dK^{(j)} += (dS_i^{(j)})^\top Q_i$
  - **end for**
  - 将 $dK^{(j)}$ 和 $dV^{(j)}$ 写入全局内存

**end for**

返回 dQ, dK, dV。

在作业的下一部分，我们将探索使用多个 GPU 训练语言模型的方法，重点关注数据并行。我们将先介绍 PyTorch 中的分布式通信基础知识，然后研究分布式数据并行训练的朴素实现，再实现和基准测试各种改进以提高通信效率。

### 2.1 PyTorch 中的单节点分布式通信

让我们先看一个 PyTorch 中的简单分布式应用，目标是生成四个随机整数张量并计算它们的和。

在下面的分布式场景中，我们将生成四个工作进程，每个进程生成一个随机整数张量。为了在工作进程间对这些张量求和，我们将调用 all-reduce 集合通信操作，该操作将每个进程上的原始数据张量替换为 all-reduce 的结果（即求和）。

```python
import os
import torch
import torch.distributed as dist
import torch.multiprocessing as mp

def setup(rank, world_size):
    os.environ["MASTER_ADDR"] = "localhost"
    os.environ["MASTER_PORT"] = "29500"
    dist.init_process_group("gloo", rank=rank, world_size=world_size)

def distributed_demo(rank, world_size):
    setup(rank, world_size)
    data = torch.randint(0, 10, (3,))
    print(f"rank {rank} data (before all-reduce): {data}")
    dist.all_reduce(data, async_op=False)
    print(f"rank {rank} data (after all-reduce): {data}")

if __name__ == "__main__":
    world_size = 4
    mp.spawn(fn=distributed_demo, args=(world_size, ), nprocs=world_size, join=True)
```

运行后输出如下。如预期，每个工作进程最初持有不同的数据张量。all-reduce 操作后，data 在每个工作进程上被原地修改为 all-reduce 的结果。[^2]

[^2]: 如果你多次运行此脚本，你会注意到打印输出的顺序不确定。由于此应用在分布式设置中运行，我们无法控制命令执行的确切顺序——我们唯一的保证是在 all-reduce 操作完成后，各进程将持有按位相同的结果张量。

```
$ uv run python distributed_hello_world.py
rank 3 data (before all-reduce): tensor([3, 7, 8])
rank 0 data (before all-reduce): tensor([4, 4, 7])
rank 2 data (before all-reduce): tensor([6, 0, 7])
rank 1 data (before all-reduce): tensor([9, 5, 3])
rank 1 data (after all-reduce): tensor([22, 16, 25])
rank 0 data (after all-reduce): tensor([22, 16, 25])
rank 3 data (after all-reduce): tensor([22, 16, 25])
rank 2 data (after all-reduce): tensor([22, 16, 25])
```

`mp.spawn` 生成 `nprocs` 个进程来运行 `fn`。函数 `fn` 以 `fn(rank, *args)` 的形式调用，其中 rank 是工作进程的索引。每个工作进程属于一个进程组（process group），通过 `dist.init_process_group` 初始化。

在此示例中，我们使用"gloo"后端初始化进程组，但也有其他后端可用。特别是"nccl"后端将使用 NVIDIA NCCL 集合通信库，对 CUDA 张量通常更高效。经验法则：GPU 分布式训练用 NCCL，CPU 分布式训练和/或本地开发用 Gloo。

运行多 GPU 任务时，确保不同 rank 使用不同的 GPU。一种方法是在 setup 函数中调用 `torch.cuda.set_device(rank)`。

**术语：**
- **节点（node）**：网络上的一台机器。
- **工作器（worker）**：参与分布式训练的程序实例。
- **世界大小（world size）**：进程组中的工作器总数。
- **全局 rank（global rank）**：唯一标识进程组中工作器的整数 ID（0 到 world_size-1）。
- **本地世界大小（local world size）**：在给定节点上本地运行的工作器数量。
- **本地 rank（local rank）**：唯一标识机器上本地工作器索引的整数 ID（0 到 local_world_size-1）。

### 2.2 分布式数据并行训练的朴素实现

### 2.4 4D 并行化

虽然实现更复杂，但我们可以沿更多轴并行化训练过程。最常讨论的5种并行方法：

- **数据并行（DP）**——数据批次在多个设备间拆分，每个设备为自己的批次计算梯度。
- **完全分片数据并行（FSDP）**——优化器状态、梯度和权重在设备间拆分。
- **张量并行（TP）**——激活值沿新维度分片，每个设备计算其分片的输出结果。
- **流水线并行（PP）**——模型按层拆分为多个阶段，每个阶段在不同设备上运行。
- **专家并行（EP）**——将专家（在混合专家模型中）分到不同设备。

通常，我们总是组合 FSDP 和 TP，因此可以将它们视为单个并行化轴。这给我们留下4个并行化轴：DP、FSDP/TP、PP 和 EP。

参见 TPU Scaling Book 第5部分（Austin et al. [2025]）了解更多详情。关于流水线并行的更详细讨论，参见 The Ultra-Scale Playbook 附录（Nouamane Tazi [2025]）。

## 3 优化器状态分片（Optimizer State Sharding）

分布式数据并行训练概念简单且通常非常有效，但要求每个 rank 持有模型参数和优化器状态的独立副本。这种冗余可能带来显著的内存开销。例如，AdamW 优化器每个参数维护两个浮点数，意味着它消耗的内存是模型权重的两倍。Rajbhandari et al. [2020] 描述了几种通过在 rank 间划分(1)优化器状态、(2)梯度和(3)参数来减少数据并行训练中冗余的方法。

在作业的这部分，我们将通过实现简化版的优化器状态分片来减少每个 rank 的内存消耗。每个 rank 的优化器实例只处理参数的一个子集（约 1/world_size）。当每个 rank 的优化器执行优化器步骤时，它只更新其分片中的模型参数子集。然后每个 rank 将其更新后的参数广播给其他 rank，以确保优化器步骤后模型参数保持同步。

**问题 (optimizer_state_sharding)：15分**

实现一个处理优化器状态分片的 Python 类。该类应包装任意 PyTorch `optim.Optimizer`，负责在每个优化器步骤后同步更新的参数。推荐的公共接口：

- `def __init__(self, params, optimizer_cls: Type[Optimizer], **kwargs: Any)`：初始化分片状态优化器。`params` 是要优化的参数集合；这些参数将在所有 rank 间分片。`optimizer_cls` 指定要包装的优化器类型（如 `optim.AdamW`）。剩余关键字参数转发给 `optimizer_cls` 的构造函数。确保在此方法中调用 `torch.optim.Optimizer` 超类构造函数。
- `def step(self, closure, **kwargs)`：调用被包装优化器的 `step()` 方法。更新参数后与其他 rank 同步。
- `def add_param_group(self, param_group: dict[str, Any])`：向分片优化器添加参数组。
